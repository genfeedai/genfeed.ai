## Session 1 — Fix Sentry-detected slow API requests (auth hot path, articles slug, elements)

### System flow (auth hot path, after)

```
Request with Bearer JWT
  → CombinedAuthGuard → BetterAuthGuard → BetterAuthStrategy.validate
      → BetterAuthService.verifyToken
          → in-memory claims cache (hit: no JWKS DB read, no crypto)
          → miss: instance.api.verifyJWT → cache claims until min(60s, exp)
      → BetterAuthIdentityResolverService.resolve(sub)
          → BetterAuthIdentityCacheService.get (Redis, ba-identity:{userId}, TTL 120s)
          → hit: return identity (0 DB queries)
          → miss: resolveFromDatabase (users/members/orgs/brands) → cache.set
  → RequestContextMiddleware (already Redis-cached, unchanged)

Invalidation (org/brand switch, onboarding, member invite):
  usersService.patch(lastUsedOrganizationId / lastUsedBrandId)
  → Promise.all([ rcCache, accessBootstrapCache, betterAuthIdentityCache ].invalidateForUser)
```

### Key decisions

- **Identity cache TTL 120s** — org/brand switches invalidate eagerly; TTL only bounds staleness for uninstrumented mutations (e.g. member deactivation).
- **Claims memoization is safe** — `verifyJWT` checks only signature + expiry (no revocation), so caching claims ≤ token exp changes no security semantics.
- **Committed regenerated `enum-field-map.ts`** — master's copy was stale (SocialConversation/SocialMessage missing); `check:model-metadata` CI gate would fail otherwise.
- **Skipped Stripe-webhook invalidation sites** — subscription changes don't alter identity (org/brand/superadmin).

## Session 2 — CI fail-fast: cancel doomed full-suite runs

### System flow

```
docker-publish.yml (release/dispatch)
├── verify-suite ──> full-suite.yml
│                    ├── ci (ci.yml: lint/typecheck/test shards)   ─┐
│                    ├── build-verify (prod image boot check)       ─┤ needs
│                    ├── e2e (needs ci+build-verify, skipped if red)│
│                    └── cancel-doomed-run (NEW)  <─────────────────┘
│                        if: failure() → gh run cancel <run_id>
└── verify-selfhosted (14.5-min Docker boot check, SIBLING —
                       killed by the cancel because run_id is shared)
```

### Affected components

- CI only: `.github/workflows/full-suite.yml`, `docker-publish.yml`, `deploy-ecs.yml`

### What was done

- [x] Diagnosed run 28641997045: first test failure at 06:09:38, all tests done 06:14, run dragged to 06:22:36 solely on the passing-but-pointless Self-Hosted Build Verify sibling
- [x] Added `cancel-doomed-run` job to `full-suite.yml`: `needs: [ci, build-verify]`, `if: failure()`, runs `gh run cancel ${{ github.run_id }}`
- [x] Granted `actions: write` on both caller jobs (`verify-suite` in docker-publish.yml, `qa-before-deploy` in deploy-ecs.yml) — reusable workflows cannot elevate past the caller's job grant
- [x] Verified all three files with actionlint (clean)

### Key decisions

- **No run-level instant fail-fast** — the run had 5 independent failing jobs; instant cancel would surface one failure per iteration. Watchdog waits for the full CI gate (`needs` semantics), so every shard reports before the cancel → zero failure-visibility loss, only the doomed tail is cut (~8–13 min per red publish/deploy run).
- **Kept `fail-fast: false` on ci.yml shard matrices** — deliberate see-all-failures behavior, correct for the fix-everything-in-one-pass loop.
- **Rejected serializing `verify-selfhosted` behind `verify-suite`** — would add the whole suite duration to every GREEN publish.
- **Known trade-off:** cancelled runs conclude "cancelled" not "failure"; failed jobs still show red. Any alerting on `conclusion == failure` for docker-publish/deploy-ecs must also match `cancelled`.

### Files changed

- `.github/workflows/full-suite.yml` — new `cancel-doomed-run` job
- `.github/workflows/docker-publish.yml` — `actions: write` on `verify-suite`
- `.github/workflows/deploy-ecs.yml` — `actions: write` on `qa-before-deploy`

### Next steps

- Watch the next red release run to confirm the cancel fires after the CI gate completes and kills the self-hosted verify sibling.
