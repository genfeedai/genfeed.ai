### Key decisions

- **No run-level instant fail-fast** — the run had 5 independent failing jobs; instant cancel would surface one failure per iteration. - **Known trade-off:** cancelled runs conclude "cancelled" not "failure"; failed jobs still show red. Any alerting on `conclusion == failure` for docker-publish/deploy-ecs must also match `cancelled`.

## Session 4 — Agent guardrails: deterministic hooks + nested CLAUDE.md (model Fable 5 high)

Vincent asked what guardrails would stop agents repeating the same mistakes and cut token cost. ### System flow

```
Bash tool call
  → PreToolUse hook: .claude/hooks/pre-bash-guard.py (stdin JSON, exit 2 = block, stderr → agent)
      ├── git push targeting master/main (token-exact refspec match, incl. HEAD:master; bare push while on trunk)
      ├── npm/npx/yarn/pnpm in command position (bun-only)
      ├── unscoped `bun test` / `turbo run test` without --filter (full suites → CI)
      ├── `vercel` when .vercel/project.json missing
      └── `git add`/`git commit` touching .env* (checks staged files on commit)
Edit/Write tool call
  → PostToolUse hook: .claude/hooks/post-edit-biome.sh
      → bunx biome check --write <file> (ts/tsx/js/jsx/json only)
      → auto-fixes in place; error-level issues → exit 2, stderr → agent fixes immediately
CLAUDE.md loading
  → nested files load only when working in that dir:
```

## Session 6 — #1244 command palette double-registration fix (model Fable 5 high)

Fixed Sentry `APP-GENFEED-AI-3` ("Commands already registered" warnings on `/settings`) at the registration/cleanup level. Branch `fix/1244-command-palette-double-registration`, worktree `funny-newton-d0bb54`.

### System flow

```
CommandPaletteInitializer (mounted per layout branch in app-protected-layout.tsx)
└── useDefaultCommandsRegistration (packages/hooks)
    ├── mount: registerDefaultCommands(org, brand)      ← now returns registered ids
    │   └── CommandPaletteService.registerCommands(...)  ← now returns string[] of ids it actually added
    └── cleanup (NEW): CommandPaletteService.unregisterCommands(ids)  ← new bulk method
        → remount (StrictMode / layout-branch swap / /settings nav) registers into a clean slate → no warn
```

### Root cause

`CommandPaletteService` is a module singleton whose command Map outlives React components. `useDefaultCommandsRegistration` registered ~25 default commands in a `useEffect` keyed on `[orgSlug, brandSlug]` with **no cleanup**, so any remount of `CommandPaletteInitializer` re-registered every id and hit the service's already-registered warn path. The sibling hooks (`useBrandCommands`, `useAdminCommandRegistration`, `useAgentThreadCommands`) all had cleanup — only the defaults hook was missing it.

### What was done

- [x] `command-palette.service.ts`: `registerCommands` returns the ids it actually registered (skipped dupes excluded); new bulk `unregisterCommands(ids)` (single info log, error log for missing ids).
- [x] `commands.registry.ts`: `registerDefaultCommands` passes the registered ids through.
- [x] `use-default-commands-registration.ts`: effect cleanup unregisters exactly the ids it registered; failure path unregisters nothing.
- [x] Tests: service (return values, bulk unregister, register→unregister→register remount cycle asserts no warn + each command exactly once), registry (ids passthrough), hook (unmount unregisters, remount re-registers with no leftovers, brand-change swap, failed-registration no-op). Updated `app-protected-layout.test.tsx` service mock to the new contract.

### Key decisions

- **Cleanup owns exactly what it registered** — `registerCommands` returning only newly added ids means a partially skipped registration never unregisters commands owned by another registrant.
- **Kept the already-registered warn** — with cleanup in place it stays a genuine-bug signal instead of routine noise; no idempotent-overwrite masking.
- **No layout restructure** — initializer stays mounted per layout branch; #1231 (settings IA) can reshape layouts later without re-breaking this.
