### Key decisions

1. **Fresh worktrees need manual provisioning**: `bun install` (worktrees inherit no `node_modules`) + `bun run --filter=@genfeedai/prisma db:generate` (root `postinstall` only builds skill symlinks — it does NOT generate the Prisma client).
2. **Env vars reach Nest only when the API runs directly** (`cd apps/server/api && bun run dev`). Running through `bunx turbo run dev` **strips** any `process.env` var not declared in `turbo.json` — this silently ate an inline `BETTER_AUTH_SECRET`.
3. **DB synced with `prisma db push`, not `migrate deploy`.** The local `genfeed` DB is a `db push`-seeded skeleton: it has tables but **no `_prisma_migrations` history**, so `migrate deploy` would fail on "table already exists." Targeted `--url postgresql://genfeed@localhost:5432/genfeed` explicitly (trust auth, `genfeed` owns the tables) — the Prisma CLI's own default `DATABASE_URL` (in `packages/prisma/prisma/.env`) still points at the `local-genfeedai...rds.amazonaws.com` RDS, which must NOT be the push target.
4. **Auth config is entirely absent from committed env files** — `BETTER_AUTH_SECRET`, `BETTER_AUTH_ENABLED`, `BETTER_AUTH_TRUSTED_ORIGINS` are not in `../../.env.local`, `api/.env`, or `api/.env.local`. Provided them session-only via `process.env`. The secret was generated once and kept in the session scratchpad — never written to the repo.
5. **Same-origin is required for the session cookie.** The app hard-redirects `localhost:3000` → `local.genfeed.ai:3000`, but the committed frontend env calls the API cross-site at `localhost:3010`, so the Better-Auth cookie never sticks. Fix: `BETTER_AUTH_URL=http://local.genfeed.ai:3000` + `NEXT_PUBLIC_API_ENDPOINT=http://local.genfeed.ai:3000/v1` so everything flows through the app's `/v1` proxy on one origin.
6. **Magic-link completion**: `auth_tokens` stores a **hashed** token, so the DB row can't reconstruct the link. Temporarily logged the raw magic-link URL in `better-auth-mailer.service.ts` (dev only), used it, then **reverted** the edit. Session valid through **2026-07-12**.
7. **Login heading fix (shipped)**: the magic-link/password/reset views capped their card at `max-w-[320px]`, narrower than the chooser's `max-w-sm`, so long headings wrapped ("Sign in with a magic link" needs 337px, "Sign in with email and password" 447px at `text-3xl`). Unified all auth cards to `max-w-sm` and dropped the shared `AuthHeader` h1 to `text-2xl` → every heading one line. **PR #1303.**
8. **This handoff landed off `master`**, separate from the login-fix branch, since the runtime setup work isn't part of that scoped PR.

## Session 2 — Platform admin settings (model-cost margin) — issue #1278

### Context

Build the *real* platform-scoped admin settings surface under `/admin` (the org-scoped
`/settings/admin` stub was removed earlier). First knob: the **model-cost margin
multiplier** — the markup Genfeed applies on provider model costs when billing customers.
Platform-wide, distinct from org/personal/brand settings, gated to platform superadmins.
Worktree `goofy-diffie-d20056` off `master`. Shipped as PR **#1333** → `master`.

### System flow

```text
Operator (superadmin) → /admin/administration/platform-settings (page.tsx + content client)
  → AdminPlatformSettingsService.getSettings()  GET /admin/platform-settings
      → PlatformSettingsController (@UseGuards IpWhitelistGuard, SuperAdminGuard)
      → PlatformSettingsService.getSingleton()  (race-safe: findOne key='platform' → create default → re-read on unique race)
  → operator edits marginMultiplier → PATCH /admin/platform-settings (only marginMultiplier accepted)
      → updateSingleton(): patch whitelisted field → setRuntimeMarginMultiplier(value) in THIS API process

Margin → customer price (authoritative path):
  workers CronModelWatcher.discoverNewModels()
    → Step 0: PlatformMarginService.hydrate()  (reads platformSetting.marginMultiplier → setRuntimeMarginMultiplier)
    → fal / hugging-face / replicate discovery bake Model.cost via applyMargin(cost)  (default arg = runtime multiplier)
  API process also hydrates on boot (OnModuleInit) + on every update → covers runtime fal-provider estimates
```

### Affected components

- **Data** (`packages/prisma`): new `PlatformSetting` singleton model (unique `key` sentinel `'platform'`, `marginMultiplier Float @default(1.0)`), migration `20260705000000_add_platform_settings`, regenerated `enum-field-map.ts` metadata map.
- **Backend** (`apps/server/api`): `collections/platform-settings/` (service+entity+schema+DTOs) and `endpoints/admin/platform-settings/` (controller+module, `SuperAdminGuard` + `IpWhitelistGuard`), registered in `admin.module.ts`.
- **Pricing** (`packages/pricing` + `packages/helpers`): `applyMargin(cost, multiplier?)` scales sell price; new process-scoped `setRuntimeMarginMultiplier`/`getRuntimeMarginMultiplier`.
- **Workers** (`apps/server/workers`): `PlatformMarginService` (DB → runtime hydrate), wired into the model-discovery cron; added `@genfeedai/constants` dep.
- **Shared**: serializer triplet (`packages/serializers`), `IPlatformSetting`/`IUpdatePlatformSettingPayload` (`packages/interfaces`), `PLATFORM_SETTING_KEY` (`packages/constants`).
- **Frontend** (`apps/app`): `admin/administration/platform-settings/` page pair + nav entry + `AdminPlatformSettingsService`.

### What was done

- [x] All 3 acceptance criteria of #1278 (view/set margin in /admin; margin applied in cost→price; superadmin-only).
- [x] Colocated tests: applyMargin multiplier + runtime setter, singleton semantics + key-smuggle guard, worker hydration, DTO validation.
- [x] Focused verification (type-check api/workers/app/services/prisma, serializers/prisma suites, biome, model-metadata check) + full CI green (29 pass).

### Key decisions

1. **Process-scoped runtime margin multiplier** in `@genfeedai/pricing` (mutable module state hydrated per process) instead of threading a param through every `applyMargin` call site. Chosen because callers span sync/async across 3 workers discovery services — a missed site = inconsistent margins between providers. Safe because `applyMargin` has **no frontend/SSR callers** (all backend). `applyMargin`'s default param reads the runtime value at call time; setter + fn both sanitize non-finite/≤0 → 1.0.
2. **Authoritative application point = model-discovery (workers)**, which bakes `Model.cost`. API-side hydration (boot + on update) covers only the secondary runtime `fal-provider` estimate. Margin thus takes effect on the next discovery run, and immediately in the API instance that saved it.
3. **Singleton via unique `key` sentinel** + race-safe `getSingleton` (create-if-missing, re-read on unique conflict). Mirrors no existing singleton (Setting is per-user, OrganizationSetting per-org).
4. **Data in `collections/`, admin controller in `endpoints/admin/`** gated by `SuperAdminGuard` + `IpWhitelistGuard` — mirrors the announcements admin pattern exactly.
5. **Update DTO exposes only `marginMultiplier`** (not `PartialType(Create)`), and `updateSingleton` whitelists that field — the singleton `key` can never be renamed/orphaned via the API (Codex-flagged, see below).
