## Session 2 — Expose clip projects on MCP (#1245)

### Context

Issue #1245 (child of clips epic #1234, Phase 3): give an MCP/AI client the full
clip-project lifecycle so it can create → review highlights → generate → read results
end-to-end, matching the API and frontend. Tri-surface parity — this is the MCP surface.

Dependency audit up front (never trust the epic body): #1235 (trim primitive) and #1236
(raw-cut generation mode, service layer) are **closed**; #1238 (expose `mode` on the
generate API), #1237, #1239 are **open**. So the live `generate` endpoint is still
avatar-only — `GenerateClipsDto` has no `mode`, `avatarId`/`voiceId` are required — even
though `ClipGenerationService` already branches on `mode` internally. Per the task brief:
expose the avatar path now, structure the tool so raw-cut slots in with no rewrite.

  ▼

  │  ├─ writes (analyze/create/generate) ─► APPROVAL_REQUIRED_TOOLS ─► createApproval (pending)

  │  └─ reads ─────────────────────────────► executeTool directly

classify(name) === 'clip-projects' ─► handleClipProjectsTool(client, name, args)
  ▼
ClientService.<method> ─► ClipsClient ─► BaseApiClient (bearer) ─► live clip-projects API

### Context

`/settings/admin` stub was removed earlier). First knob: the **model-cost margin

Worktree `goofy-diffie-d20056` off `master`. Shipped as PR **#1333** → `master`.

### System flow

```text
Operator (superadmin) → /admin/administration/platform-settings (page.tsx + content client)
  → AdminPlatformSettingsService.getSettings()  GET /admin/platform-settings
      → PlatformSettingsController (@UseGuards IpWhitelistGuard, SuperAdminGuard)
      → PlatformSettingsService.getSingleton()  (race-safe: findOne key='platform' → create default → re-read on unique race)
  → operator edits marginMultiplier → PATCH /admin/platform-settings (only marginMultiplier accepted)
      → updateSingleton(): patch whitelisted field → setRuntimeMarginMultiplier(value) in THIS API process

Margin → customer price (authoritative path):
  workers CronModelWatcher.discoverNewModels()
    → Step 0: PlatformMarginService.hydrate()  (reads platformSetting.marginMultiplier → setRuntimeMarginMultiplier)
    → fal / hugging-face / replicate discovery bake Model.cost via applyMargin(cost)  (default arg = runtime multiplier)
  API process also hydrates on boot (OnModuleInit) + on every update → covers runtime fal-provider estimates
```

### What was done

- [x] All 3 acceptance criteria of #1278 (view/set margin in /admin; margin applied in cost→price; superadmin-only).
- [x] Colocated tests: applyMargin multiplier + runtime setter, singleton semantics + key-smuggle guard, worker hydration, DTO validation.
- [x] Focused verification (type-check api/workers/app/services/prisma, serializers/prisma suites, biome, model-metadata check) + full CI green (29 pass).

### Key decisions

1. **Process-scoped runtime margin multiplier** in `@genfeedai/pricing` (mutable module state hydrated per process) instead of threading a param through every `applyMargin` call site. Chosen because callers span sync/async across 3 workers discovery services — a missed site = inconsistent margins between providers. Safe because `applyMargin` has **no frontend/SSR callers** (all backend). `applyMargin`'s default param reads the runtime value at call time; setter + fn both sanitize non-finite/≤0 → 1.0.
2. **Authoritative application point = model-discovery (workers)**, which bakes `Model.cost`. API-side hydration (boot + on update) covers only the secondary runtime `fal-provider` estimate. Margin thus takes effect on the next discovery run, and immediately in the API instance that saved it.
3. **Singleton via unique `key` sentinel** + race-safe `getSingleton` (create-if-missing, re-read on unique conflict). Mirrors no existing singleton (Setting is per-user, OrganizationSetting per-org).
4. **Data in `collections/`, admin controller in `endpoints/admin/`** gated by `SuperAdminGuard` + `IpWhitelistGuard` — mirrors the announcements admin pattern exactly.
5. **Update DTO exposes only `marginMultiplier`** (not `PartialType(Create)`), and `updateSingleton` whitelists that field — the singleton `key` can never be renamed/orphaned via the API (Codex-flagged, see below).
