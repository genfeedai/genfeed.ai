### System flow — how a backend app's type-check resolves

```
package.json  "type-check": "tsc --noEmit -p tsconfig.typecheck.json"
        │
        ▼
apps/server/<app>/tsconfig.typecheck.json   (3-line: extends base + include/exclude)
        │  extends
        ▼
apps/server/tsconfig.typecheck.base.json    (NO baseUrl · module ESNext ·
        │                                     moduleResolution bundler ·
        │                                     useDefineForClassFields false ·
        │                                     full @alias path superset)
        │  paths resolve relative to base dir (apps/server/)  → @api/* → ./api/src/*
        ▼
turbo run type-check  (dependsOn ^build → deps' dist built first)
        │
        ├─ api/turbo.json      inputs += files/src        (cross-app coupling)
        ├─ workers/turbo.json  inputs += api/src,files/src (cross-app coupling)
        └─ root turbo.json     inputs += base config       (compiler-option changes bust all)

Runtime path UNTOUCHED:  start:prod → node -r tsconfig-paths/register → reads
tsconfig.json (baseUrl+paths) — never the typecheck config → zero runtime regression.
```

### Key decisions

| Decision | Rationale |
|---|---|
| Separate `tsconfig.typecheck.json`, NOT rework runtime `tsconfig.json` | Webpack inlines internal aliases at build time → `tsconfig-paths/register` is ~vestigial; touching the runtime config risks `start:prod`. Inverting #1145's "separate runtime tsconfig" hint is strictly safer. |
| `useDefineForClassFields: false` | Runtime is swc with `legacyDecorator`+`decoratorMetadata` (NestJS legacy model; `workers/tsconfig.app.json` already sets it false). The checker must mirror runtime field semantics, not impose stricter ones. Removed all 29 TS2612 without per-file `declare` churn. |
| Exclude specs from the app type-check | Matches the proven `ee-billing` config; 1201 of api's 1272 errors were stale test-mock drift. `import.meta` (trap #3) still handled because non-spec src (`main.ts` etc.) uses it → ESNext required regardless. |
| Ship 10/12; `notifications`+`files` as follow-ups | Issue explicitly sanctions api-first + tracked follow-ups. Both remaining apps need type *design* work (discord.js payload types; request-body DTOs) with behavior-change risk — better as focused PRs. |
| Keep ee-billing `--filter` shim + inputs | api's task checks the OSS billing flavor; ee-billing validates the EE contract. `--affected` won't select ee-billing on api-only changes (not a graph dependent). Not subsumed. |
