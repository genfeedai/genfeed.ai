# TrainingFilterUtil

A utility class that provides consistent, reusable lookup builders for training-related ingredient queries.

## Problem Solved

Before this utility, the trainings controller had repeated lookup patterns for finding training-related ingredients:

- Source images lookup: finds ingredients by source IDs (training.sources → ingredients)
- Generated images lookup: finds ingredients by metadata/model (training.model → metadata → ingredients)

These patterns were easy to forget required filters (like `isDeleted: false`), leading to bugs where soft-deleted ingredients appeared in results.

This led to:

- 🔴 **Code duplication** - Same lookup logic repeated in multiple places
- 🔴 **Missing filters** - Easy to forget `isDeleted` filter, causing bugs
- 🔴 **Maintenance burden** - Bug fixes needed to be applied multiple times
- 🔴 **Inconsistency** - Different implementations could have different filters

## Solution

`TrainingFilterUtil` centralizes all training-specific lookup patterns into a single utility class with all required filters included.

## Features

### Lookup Builders

#### 1. `buildSourceImagesLookup(options)`

Creates complete $lookup stage for finding ingredient source images for a training.

Includes all required filters:

- User match (ensures user owns the ingredients)
- Category filter (IMAGE by default)
- ID match (finds ingredients in training.sources array)
- isDeleted filter (excludes soft-deleted ingredients)

```typescript
const sourceLookup = TrainingFilterUtil.buildSourceImagesLookup({
  sourceIdsVar: '$sources',
  userIdVar: '$user',
  as: 'sourceImages',
});
// Returns: Complete $lookup stage with all filters
```

**Options:**

- `sourceIdsVar` - Variable name for source IDs array (e.g., '$sources')
- `userIdVar` - Variable name for user ID (e.g., '$user')
- `as` - Output field name (default: 'sourceImages')
- `category` - Ingredient category (default: IngredientCategory.IMAGE)

#### 2. `buildGeneratedImagesLookup(options)`

Creates complete $lookup stage for finding ingredients generated by a training model.

Includes all required filters:

- Category filter (IMAGE by default)
- Metadata match (finds ingredients with matching metadata.model)
- isDeleted filter (excludes soft-deleted ingredients)

```typescript
const generatedLookup = TrainingFilterUtil.buildGeneratedImagesLookup({
  metadataIdsVar: '$metadataWithModel._id',
  as: 'generatedImages',
});
// Returns: Complete $lookup stage with all filters
```

**Options:**

- `metadataIdsVar` - Variable name for metadata IDs array (e.g., '$metadataWithModel.\_id')
- `as` - Output field name (default: 'generatedImages')
- `category` - Ingredient category (default: IngredientCategory.IMAGE)

## Usage Examples

### Before (Trainings Controller - Duplicated Logic)

```typescript
// Source images lookup - easy to forget filters
{
  $lookup: {
    from: 'ingredients',
    let: {
      sourceIds: '$sources',
      userId: '$user',
    },
    pipeline: [
      {
        $match: {
          $expr: {
            $and: [
              { $eq: ['$user', '$$userId'] },
              { $eq: ['$category', IngredientCategory.IMAGE] },
              { $in: ['$_id', '$$sourceIds'] },
              // Forgot isDeleted filter! ❌
            ],
          },
        },
      },
    ],
    as: 'sourceImages',
  },
}

// Generated images lookup - easy to forget filters
{
  $lookup: {
    from: 'ingredients',
    let: {
      metadataIds: '$metadataWithModel._id',
    },
    pipeline: [
      {
        $match: {
          $expr: {
            $and: [
              { $eq: ['$category', IngredientCategory.IMAGE] },
              { $in: ['$metadata', '$$metadataIds'] },
              // Forgot isDeleted filter! ❌
            ],
          },
        },
      },
    ],
    as: 'generatedImages',
  },
}
```

### After (Trainings Controller - Clean & DRY)

```typescript
import { TrainingFilterUtil } from '@api/helpers/utils/training-filter';

// Use utility for source images lookup
TrainingFilterUtil.buildSourceImagesLookup({
  sourceIdsVar: '$sources',
  userIdVar: '$user',
  as: 'sourceImages',
});

// Use utility for generated images lookup
TrainingFilterUtil.buildGeneratedImagesLookup({
  metadataIdsVar: '$metadataWithModel._id',
  as: 'generatedImages',
});
```

### Complete Pipeline Example

```typescript
const pipeline: PipelineStage[] = [
  {
    $match: {
      _id: new Types.ObjectId(trainingId),
    },
  },
  {
    $lookup: {
      from: 'metadata',
      let: { trainingModel: '$model' },
      pipeline: [
        {
          $match: {
            $expr: { $eq: ['$model', '$$trainingModel'] },
          },
        },
      ],
      as: 'metadataWithModel',
    },
  },
  TrainingFilterUtil.buildSourceImagesLookup({
    sourceIdsVar: '$sources',
    userIdVar: '$user',
    as: 'sourceImages',
  }),
  TrainingFilterUtil.buildGeneratedImagesLookup({
    metadataIdsVar: '$metadataWithModel._id',
    as: 'generatedImages',
  }),
  {
    $addFields: {
      totalSources: { $size: { $ifNull: ['$sourceImages', []] } },
      totalGeneratedImages: {
        $size: { $ifNull: ['$generatedImages', []] },
      },
    },
  },
];
```

## Benefits

✅ **Prevents Mistakes** - Can't forget required filters (isDeleted, category, etc.)
✅ **Training-Specific** - Follows pattern of domain-specific filter utils (Article, Preset, Template)
✅ **Consistency** - Same lookup pattern used everywhere
✅ **Maintainability** - Update filter logic in one place
✅ **Discoverability** - Training-related code in training-specific utility
✅ **Type Safety** - Full TypeScript support
✅ **Documentation** - Well-documented with JSDoc comments

## Pattern Consistency

This follows the established pattern:

- `ArticleFilterUtil` - Article-specific filters
- `PresetFilterUtil` - Preset-specific filters
- `TemplateFilterUtil` - Template-specific filters
- `TrainingFilterUtil` - Training-specific filters
